# 民宿 AI 内容增长工具 MVP 需求说明

## 1. 项目概述

民宿 AI 内容增长工具面向 5-10 间房的小体量民宿老板。第一版通过“粘贴 OTA 平台好评 + 上传民宿图片 -> 自动生成小红书笔记”的方式，降低老板做内容运营的门槛。

本项目独立于其他业务系统，不复用其他行业项目的业务模型、客户模型、数据结构或文档。

## 2. 用户与场景

### 2.1 目标用户

- 5-10 间房的小体量民宿老板。
- 民宿可能位于城市周边、景区附近、乡村、古镇、海边、山野等场景。
- 没有专职内容运营人员。
- 有 OTA 平台好评，但不会整理成小红书内容。
- 希望尽快获得可复制发布的笔记。

### 2.2 核心痛点

- 不知道小红书该写什么。
- 没时间持续运营内容。
- 好评分散在 OTA 平台，难以转化成传播素材。
- 不会把真实住客反馈转成有吸引力但不夸大的表达。
- 缺少发布后的效果观察和复盘。

### 2.3 核心使用路径

1. 老板进入工具。
2. 粘贴 OTA 平台好评文本。
3. 上传本次准备发布的民宿图片。
4. 可选补充民宿名称、城市、卖点、周边等信息。
5. 点击生成。
6. 系统返回小红书标题、正文、标签和图文预览。
7. 老板复制到小红书发布。
8. 后续系统通过公开搜索回访发布效果。

交互原则：

- 老板只管丢素材，系统负责理解、整理、写作和提示风险。
- 好评文本是最低必填项；图片和民宿档案是增强项。
- 不因为档案不完整阻断生成，但未提供的信息不能写成确定事实。

## 3. MVP 目标

### 3.1 必须达成

- 让非运营型民宿老板在 1-2 分钟内得到一篇可复制发布的小红书笔记。
- 生成内容有明确种草感，但不过度营销。
- 生成内容基于老板输入和好评文本，不编造确定事实。
- 记录每次生成内容，用于后续偏好学习和发布回访。

### 3.2 暂不追求

- 不做截图 OCR。
- 不做自动发布到小红书。
- 不做大规模小红书采集。
- 不做复杂多账号运营后台。
- 不做完整 CRM 或订单管理。

## 4. 功能范围

### 4.1 素材输入

第一版首页应优先让老板完成两件事：

- 粘贴 OTA 平台好评文字。
- 上传准备发布的小红书图片。

输入要求：

- 好评文字是必填项，建议至少 30 个字。
- 图片可选，最多 6 张。
- 图片用于图文预览和后续视觉模型理解；在视觉模型接入前，不得根据图片编造未识别事实。
- 不要求老板手动标注图片内容。

### 4.2 民宿档案

老板可填写并保存以下资料：

- 民宿名称
- 城市 / 景区 / 商圈
- 民宿类型，如亲子、情侣、团建、宠物友好、山景、海景、古镇等
- 主要卖点
- 适合人群
- 房间数量
- 价格区间，可选
- 小红书主页链接或账号名，可选
- 禁止夸大的信息或敏感表达，可选
- 老板偏好的内容风格，可选

使用约束：

- 未填写的信息不能在生成内容中写成确定事实。
- 系统可以把缺失信息表达为建议或可补充项，但不能自行补全为事实。
- 民宿档案在首页中属于可选补充信息，不应挡在生成路径前面。

### 4.3 好评输入

第一版只支持文本粘贴。输入来源可包括：

- 携程、飞猪、美团、途家、木鸟等 OTA 平台评论。
- 老板自行整理的住客反馈。
- 私域聊天中得到的客人评价文本。

输入要求：

- 支持单条或多条好评。
- 系统应识别高频关键词、住客感受、具体亮点和可引用表达。
- 不要求老板手动标注结构。

### 4.4 笔记生成

系统生成一份完整的小红书笔记草稿，包含：

- 3-5 个标题备选
- 1 篇正文
- 8-15 个话题标签
- 图文预览或 3-6 条配图建议
- 内容安全提醒，可选
- 生成依据摘要，可选，用于告诉老板哪些点来自好评或档案

内容风格：

- 种草型优先。
- 采用“民宿主/内容运营视角 + 游客好评佐证”。
- 不伪装成游客真实入住。
- 可以使用自然、轻量、有画面感的表达。
- 避免过分夸张、绝对化承诺和低质营销腔。

### 4.5 额度规则

- 每账号每天最多 5 次成功生成。
- 失败不扣额度。
- 重新生成算一次。
- 需要记录生成请求、生成状态、扣减结果和错误原因。

建议状态：

- pending：请求已创建
- success：生成成功并扣额度
- failed：生成失败，不扣额度
- regenerated：重新生成产生的新记录

### 4.6 生成记录

每次生成需要保存：

- 用户账号 ID
- 民宿档案 ID
- 原始好评文本
- 上传图片元数据和后续视觉摘要
- 补充输入
- 使用模型
- Prompt 版本
- 生成标题
- 生成正文
- 生成标签
- 配图建议
- 生成时间
- 是否成功
- 是否扣额度
- 错误信息
- 老板是否复制，可选
- 老板是否标记喜欢 / 不喜欢，可选

### 4.7 发布后公开搜索回访

第一阶段可做轻量回访，不做大规模采集。

系统已知：

- 民宿名称
- 城市
- 生成标题
- 正文关键句
- 标签
- 生成时间
- 老板小红书主页，可选

可通过公开搜索尝试找到老板发布的笔记，并记录：

- 小红书笔记链接
- 标题
- 发布时间
- 点赞数
- 收藏数
- 评论数
- 匹配置信度
- 回访时间

回访结果用于后续：

- 判断哪些标题更有效。
- 判断哪些卖点更适合该民宿。
- 优化标签组合。
- 给下一次生成提供偏好上下文。

## 5. 非功能要求

### 5.1 内容合规

生成内容必须遵守以下原则：

- 不编造具体事实。
- 不伪装游客真实入住。
- 不写未经证实的价格、距离、排名、奖项、销量等。
- 不使用极限词或绝对化承诺。
- 不承诺平台无法保证的结果，如必火、必爆、住了都说好。
- 对未提供信息使用模糊表达或提示补充。

### 5.2 可切换模型

模型接入层必须支持切换不同模型供应商。建议抽象为统一接口：

- generateNote(input, options)
- embedText(input)
- evaluateOutput(input, output)

大陆生产环境优先考虑稳定可接入模型，例如 GLM-5.2 / Qwen 等。GPT 可作为精品生成或灰度通道。

### 5.3 数据留存

系统需要为后续个性化生成保留：

- 民宿档案
- 生成记录
- 老板偏好
- 历史内容
- 发布回访数据
- Prompt 版本与模型版本

可考虑 PostgreSQL + pgvector + Qwen3-Embedding。

### 5.4 许可证偏好

优先选择商业友好许可证：

- Apache-2.0
- MIT

谨慎引入：

- 限制 SaaS 使用的项目
- 商业条款不清晰的项目
- 对爬取、代理或平台访问有高风险的项目

## 7. 验收标准

MVP 可按以下标准验收：

- 输入一组 OTA 好评后，系统能稳定输出结构化 JSON。
- JSON 包含标题、正文、标签、配图建议和风险提醒。
- 输出内容不伪装游客，不编造输入之外的确定事实。
- 每日 5 次成功生成限制生效。
- 失败请求不扣额度。
- 生成记录完整保存。
- 模型供应商可以通过配置切换。
- 至少有一组测试样例覆盖“信息不足时不得编造”的场景。

## 8. 下一步建议

1. 先实现 Web 原型，跑通“好评粘贴 -> 图片上传 -> 可选补充资料 -> 生成 -> 复制”。
2. 建立 Prompt 版本管理和评测样例。
3. 设计数据库迁移脚本。
4. 接入一个大陆稳定可用模型。
5. 用真实民宿好评样本进行小范围试用。
