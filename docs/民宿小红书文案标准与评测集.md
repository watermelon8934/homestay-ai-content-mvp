# 民宿小红书文案标准与评测集

## 1. 为什么需要这份标准

当前 MVP 已经跑通“OTA 好评 -> AI 生成小红书笔记”的链路，但生成质量不能只靠主观感觉调 prompt。

我们需要建立一套可持续迭代的方法：

- 用真实小红书民宿笔记建立样本库。
- 用数据表现判断哪些样本值得学习。
- 用固定评测集判断每次 prompt 修改是变好还是变差。
- 用规则约束避免编造、感谢信口吻和平台风险。

## 2. 好样本的判断原则

可以把小红书推流结果作为判断好样本的重要依据，但不能只看绝对点赞数。

原因：

- 大号天然更容易获得高曝光和高互动。
- 城市、季节、封面图、账号粉丝量、发布时间都会影响数据。
- 高点赞不一定适合小体量民宿老板复用。
- 曝光高但收藏低，可能只是标题吸睛，未必有转化价值。

因此，好样本应该采用“同类对比 + 多指标判断”。

## 3. 推荐采集的样本类型

优先采集这些公开小红书民宿笔记：

- 城市/景区民宿推荐，例如大理、丽江、杭州、莫干山、厦门等。
- 小体量民宿账号自己发布的笔记。
- 标题和正文有真实住宿卖点，不是纯图片堆砌。
- 评论区有真实咨询，例如问位置、价格、房型、能否带宠物、离景点远不远。
- 收藏数明显高于普通同类笔记。

暂不优先采集：

- 明显达人广告硬广。
- 纯低价引流。
- 与民宿真实经营无关的网红摆拍。
- 大量使用“必住、天花板、封神、闭眼冲”等高风险表达的笔记。
- 内容主要靠博主个人影响力，而不是民宿信息本身转化的笔记。

## 4. 数据指标

### 4.1 公开可见指标

这些适合第一阶段通过公开搜索或人工记录：

- 点赞数
- 收藏数
- 评论数
- 笔记发布时间
- 标题
- 标签
- 是否有明确咨询评论
- 是否有用户追问位置、价格、房型、交通、预订方式

### 4.2 老板自有后台指标

如果老板愿意提供小红书创作中心数据，可以增加：

- 曝光量
- 阅读量 / 浏览量
- 点击率
- 平均阅读时长
- 完读率
- 主页访问量
- 私信数
- 收藏率
- 评论率
- 询单数
- 实际预订线索

## 5. 好样本评分

建议用 100 分制，不只看流量。

| 维度 | 权重 | 说明 |
| --- | ---: | --- |
| 数据表现 | 35 | 点赞、收藏、评论、曝光或同类相对表现 |
| 收藏/咨询价值 | 20 | 是否引发收藏、提问、私信、预订意向 |
| 小红书表达感 | 15 | 标题、开头、分段、语气是否像真实小红书内容 |
| 卖点清晰度 | 15 | 是否让用户快速知道适合谁、亮点是什么 |
| 可复用性 | 10 | 小体量民宿老板是否能模仿，不依赖达人身份 |
| 合规安全 | 5 | 是否避免夸大、编造、极限词和虚假承诺 |

评分建议：

- 80 分以上：可作为强学习样本。
- 65-79 分：可拆解局部优点，例如标题或开头。
- 50-64 分：仅做对照样本。
- 50 分以下：作为负样本或禁区样本。

## 6. 数据归一化

如果能拿到曝光量，应优先看比率：

- 点赞率 = 点赞数 / 曝光量
- 收藏率 = 收藏数 / 曝光量
- 评论率 = 评论数 / 曝光量
- 互动率 = (点赞数 + 收藏数 + 评论数) / 曝光量
- 咨询率 = 咨询评论数或私信数 / 曝光量

如果拿不到曝光量，则做同类对比：

- 同城市对比
- 同账号体量对比
- 同发布时间窗口对比
- 同内容类型对比，例如古城民宿、亲子民宿、院子民宿

公开数据阶段，收藏和评论通常比点赞更有参考价值。

## 7. 样本库字段

建议每条样本记录这些字段：

| 字段 | 说明 |
| --- | --- |
| sample_id | 样本 ID |
| source_url | 小红书笔记链接 |
| city | 城市 |
| scene | 场景，如古城、海边、山景、亲子、情侣 |
| account_type | 民宿自营 / 达人 / 聚合号 / 不确定 |
| title | 标题 |
| body | 正文 |
| hashtags | 话题标签 |
| image_notes | 图片类型说明 |
| likes | 点赞数 |
| collects | 收藏数 |
| comments | 评论数 |
| views | 曝光或浏览，如可得 |
| publish_date | 发布时间 |
| observed_date | 记录时间 |
| score | 综合评分 |
| useful_patterns | 可学习点 |
| risk_patterns | 风险点 |
| reusable_for_mvp | 是否适合 MVP 学习 |

## 8. 文案评测集

每次修改 prompt 后，至少用固定 10 条 OTA 好评测试。

评测维度：

| 维度 | 分值 | 判断标准 |
| --- | ---: | --- |
| 小红书感 | 20 | 不像感谢信，不像客服回复，不像新闻稿 |
| 种草价值 | 20 | 能让潜在住客知道为什么值得看 |
| 事实可靠 | 25 | 不编造未提供的设施、距离、价格、景观 |
| 结构清晰 | 15 | 标题、开头、卖点、结尾清楚 |
| 可复制发布 | 10 | 老板不大改也能发 |
| 标签和配图 | 10 | 标签合理，配图建议对应真实事实 |

低于 75 分的输出，不应作为默认生成效果。

## 9. 第一批测试好评

### 测试 1：古城小院

民宿资料：

- 名称：云朵情书客栈
- 城市：大理
- 卖点：庭院院景、干湿分离、老板热情
- 周边：大理古城内

好评：

民宿环境很不错，卫生间和洗浴间是干湿分离的，老板人超好，外面有个小院子，可以坐着吹吹风，很舒服。就在大理古城内，出来就有很多好吃的。

理想方向：

- 写“古城里、吃逛方便、有小院、干湿分离”。
- 不写安静、看山、亲子、早餐、距离。
- 不用“作为主理人收到好评”。

### 测试 2：亲子放松

民宿资料：

- 名称：山间小院
- 城市：大理
- 卖点：适合亲子、窗外能看到山

好评：

房间很干净，晚上很安静，窗外能看到山，老板回复消息很及时。我们带孩子住了两晚，整体很放松，去周边逛也方便。下次来大理还会考虑这里。

理想方向：

- 可写亲子、干净、安静、窗外看山、回复及时。
- 可写“带孩子住两晚后觉得放松”。
- 不写早餐、价格、具体景点距离。

### 测试 3：服务补救

民宿资料：

- 名称：云朵情书客栈
- 城市：大理
- 卖点：老板响应快

好评：

我们来的时候出租车半路坏了，路上又堵车，打不到车。老板知道后骑电动车来接我们，真的很感谢。房间也住得舒服。

理想方向：

- 可写“服务响应快、遇到突发情况有人帮忙”。
- 不要写“免费接送”“随叫随到”。
- 不要把个案写成长期承诺。

### 测试 4：干净卫生

民宿资料：

- 名称：青瓦小筑
- 城市：杭州
- 卖点：房间清爽

好评：

房间比图片看起来还舒服，床品干净，卫生间也没有异味。我们住了一晚，睡得挺好。

理想方向：

- 可写干净、床品、卫生间无异味、睡得好。
- 不写隔音、交通、早餐。

### 测试 5：朋友出游

民宿资料：

- 名称：海边白房子
- 城市：厦门
- 卖点：适合朋友出游

好评：

我们四个朋友一起住，公共区域很适合聊天，晚上回来还能一起点外卖。房东给的入住指引很清楚。

理想方向：

- 可写朋友出游、公共区域、入住指引清楚。
- 不写海景，除非资料或好评提供。

### 测试 6：宠物友好

民宿资料：

- 名称：松果院子
- 城市：莫干山
- 卖点：可带宠物

好评：

带狗狗来住很方便，院子里可以活动，老板提前提醒了入住注意事项。整体很省心。

理想方向：

- 可写宠物友好、院子、提前提醒、省心。
- 需要提醒发布前确认宠物政策仍然有效。

### 测试 7：情侣短住

民宿资料：

- 名称：湖边小屋
- 城市：杭州
- 卖点：适合情侣短住

好评：

房间布置很温馨，晚上灯光很好看，适合两个人慢慢待着。拍照也挺出片。

理想方向：

- 可写情侣、温馨、灯光、拍照。
- 不写湖景，除非好评或资料明确提供。

### 测试 8：交通方便但不能夸大

民宿资料：

- 名称：巷口民宿
- 城市：成都
- 周边：老街附近

好评：

位置挺方便，走出去吃东西选择很多，晚上回来也好找。老板发的位置说明很清楚。

理想方向：

- 可写吃东西方便、好找、位置说明清楚。
- 不写具体距离地铁、机场、景点。

### 测试 9：景观明确

民宿资料：

- 名称：望山居
- 城市：丽江
- 卖点：窗外能看到山

好评：

早上拉开窗帘能看到山，天气好的时候很漂亮。房间里待着也很舒服。

理想方向：

- 可写窗外看山、天气好时漂亮、房间舒服。
- 不写雪山、日照金山，除非输入明确。

### 测试 10：信息不足

民宿资料：

- 名称：木棉小院
- 城市：泉州

好评：

整体不错，老板人很好，下次还会来。

理想方向：

- 生成应更克制。
- 提醒信息不足，建议老板补充位置、房间、周边、图片等。
- 不强行写大量具体卖点。

## 10. Prompt 迭代记录

### v0.1

方向：民宿主视角 + 好评佐证。

问题：

- 容易写成感谢信。
- 常出现“作为主理人，收到这样的住客好评”。
- 容易复述“客人说：”。

### v0.2

方向：民宿老板使用好评做小红书种草宣传。

改动：

- 开头改为挑民宿场景。
- 好评作为佐证，不作为全文主结构。
- 禁止感谢信口吻。
- 加强事实约束。

待验证：

- 是否仍然擅自加入“安静、干净、景观”等常见民宿卖点。
- 标题是否能更像小红书但不夸张。
- 是否需要引入样本库做 few-shot 示例。
