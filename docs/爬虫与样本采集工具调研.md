# 爬虫与样本采集工具调研

## 1. 当前结论

MVP 阶段可以继续用爬虫，但目标必须收窄：

```text
不是做小红书数据平台，
而是小规模收集公开民宿笔记样本，用来校准 Prompt 和评测生成质量。
```

当前推荐方案：

- 第一优先：Crawlee + Playwright 做轻量样本采集器。
- 第二优先：参考 GitHub 上的小红书采集项目的字段设计和页面定位思路。
- 暂不直接引入重型小红书爬虫框架，尤其是不清楚许可证、强依赖 Cookie、强调反检测或批量下载的项目。

本项目本地依赖中已经存在：

- Crawlee 3.17.0，Apache-2.0
- Playwright 1.61.1，Apache-2.0

这两个许可证商业友好，适合作为 MVP 阶段的采集基础。

## 2. 安全和合规边界

允许：

- 采集公开可见的小红书笔记页面信息。
- 采集标题、正文、标签、点赞/收藏/评论数等公开互动数据。
- 记录笔记链接、账号类型判断、图片结构说明。
- 小规模人工触发采集，例如每次 20-50 条。
- 为 Prompt 调优做样本拆解、模式归纳和评分。

不允许：

- 导出、保存或提交用户的小红书 Cookie、Token、LocalStorage。
- 采集私信、通讯录、关注列表、个人隐私信息。
- 绕过验证码、登录限制、风控限制或付费限制。
- 高频请求、批量刷接口或模拟异常用户行为。
- 自动点赞、收藏、评论、关注、发布或私信。
- 大规模建设小红书内容数据库。

遇到验证码、登录失效、访问受限时，采集器必须停止，由用户人工处理。

## 3. 为什么不优先直接用 GitHub 小红书爬虫

GitHub 上可以搜索到不少小红书爬虫项目，但需要谨慎：

- 小红书页面和接口变化快，很多仓库维护不稳定。
- 一些项目默认要求填 Cookie 或浏览器登录态，容易把账号安全风险带进项目。
- 一些项目目标是批量下载、评论采集、接口逆向，超过本 MVP 需要。
- 许可证不一定商业友好，后续做 SaaS 时会留下隐患。

因此，GitHub 项目更适合做参考：

- 看它们如何识别笔记标题、正文、标签和互动数据。
- 看它们如何处理滚动加载和搜索结果页。
- 看它们如何定义样本字段。

不建议第一阶段直接整体引入。

## 4. 推荐采集策略

### 4.1 人工关键词 + 半自动采集

用户在小红书中搜索关键词，例如：

- 大理民宿
- 大理小院民宿
- 大理古城民宿
- 大理住宿推荐
- 民宿主理人
- 小院民宿

采集器只读取当前公开页面或用户打开的笔记页。

优点：

- 不需要导出登录 Cookie。
- 用户能人工筛掉明显不相关内容。
- 风险低，适合 MVP 验证。

缺点：

- 速度较慢，但足够收集第一批 20-50 条样本。

### 4.2 搜索结果页轻量采集

采集器从搜索结果页读取：

- 笔记标题
- 笔记链接
- 封面图文字或图片结构简述，如可得
- 可见互动数据，如可得

然后人工打开高价值笔记补正文。

适合用来建立候选池，不适合直接做 Prompt few-shot。

### 4.3 笔记详情页采集

采集器从单篇笔记页读取：

- 标题
- 正文
- 标签
- 发布时间，如可见
- 点赞、收藏、评论数，如可见
- 作者昵称
- 图片数量和大致顺序
- 评论区是否有位置、价格、房型、预订方式咨询

这是最适合 Prompt 调优的数据。

## 5. 样本字段

建议保存为 JSON：

```json
{
  "sample_id": "xhs-dali-yard-001",
  "source": "xiaohongshu",
  "source_url": "https://www.xiaohongshu.com/...",
  "observed_at": "2026-07-08",
  "query": "大理小院民宿",
  "city": "大理",
  "scene": ["小院", "古城", "慢住"],
  "account_type": "民宿自营 | 达人 | 聚合号 | 不确定",
  "title": "笔记标题",
  "body_excerpt": "正文摘录或人工摘要",
  "hashtags": ["#大理民宿"],
  "metrics": {
    "likes": null,
    "collects": null,
    "comments": null
  },
  "image_structure": [
    "首图：院子全景",
    "第二张：房间窗景",
    "第三张：周边街区"
  ],
  "comment_signals": [
    "有人问价格",
    "有人问位置"
  ],
  "useful_patterns": [
    "开头从主理人分享院子切入",
    "正文没有感谢信口吻"
  ],
  "risk_patterns": [
    "出现必住、天花板等夸张词"
  ],
  "reusable_for_prompt": true
}
```

注意：为降低版权和平台风险，样本库优先保存结构化拆解和短摘录，不保存整篇大段搬运文案；用于 few-shot 时应改写成自有示例。

## 6. 第一版采集目标

第一批只需要 20-50 条，不追求大。

建议比例：

- 10 条大理民宿自营账号笔记。
- 5 条院子型民宿笔记。
- 5 条古城/景区位置型民宿笔记。
- 5 条评论区有咨询信号的笔记。
- 5 条高互动但不适合学习的负样本。

第一批样本产出后，用来更新：

- `docs/真实民宿样本库.md`
- `docs/民宿小红书文案标准与评测集.md`
- `prompts/小红书笔记生成Prompt与JSON结构.md`
- `samples/prompt-tests/`

## 7. 技术路线

### 7.1 当前 MVP 推荐实现

使用 Crawlee + Playwright：

- 浏览器打开小红书搜索页或笔记页。
- 人工确认页面可见。
- 采集器读取可见 DOM。
- 低频滚动，不做高频请求。
- 输出 JSON 到 `samples/xhs/`。
- 人工复核后，把有价值样本整理进文案标准。

### 7.2 后续可选增强

- 增加候选样本评分。
- 增加关键词批量队列。
- 增加去重。
- 增加评论区咨询信号识别。
- 增加样本到 Prompt few-shot 的半自动转换。

## 8. 暂不做

- 不做自动登录。
- 不做 Cookie 导入导出。
- 不做接口逆向。
- 不做大规模评论采集。
- 不做自动发布或互动。
- 不做用户账号数据抓取。
