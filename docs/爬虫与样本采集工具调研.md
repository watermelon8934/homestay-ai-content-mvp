# 爬虫与样本采集工具调研

## 1. 结论

用户提到的 “fireclaw” 应为 Firecrawl。

Firecrawl 是目前 GitHub 上非常热门的 AI 爬取/网页转 Markdown 工具，适合快速把公开网页转成 LLM 可读内容，也支持搜索、抓取、结构化抽取、站点 crawl 和 MCP。

但它不一定是“小红书样本采集”的唯一最优解。

对本项目而言，推荐路线：

1. 第一优先试用 Firecrawl：用于公开网页、搜索结果、可访问笔记页的快速抽取。
2. 备用方案使用 Crawlee + Playwright：用于更复杂的动态网页和需要浏览器行为的页面。
3. 暂不做大规模小红书采集，只做“人工指定关键词/民宿/链接 -> 小批量公开样本采集”。
4. 所有采集只用于文案结构学习和评测，不采集私人数据，不做自动互动，不做绕过登录或风控。

## 2. 工具对比

| 工具 | 适合场景 | 优点 | 风险/限制 | 许可证 |
| --- | --- | --- | --- | --- |
| Firecrawl | 公开网页转 Markdown、结构化抽取、AI Agent 联网检索 | 上手快，输出适合 LLM，支持 API/MCP/云服务 | 自托管主项目 AGPL-3.0；对小红书这类强登录/强反爬平台不一定稳定 | AGPL-3.0，部分 SDK/UI 为 MIT |
| Crawlee | Node.js/TypeScript 爬虫、Playwright/Puppeteer/HTTP 混合抓取 | Apache-2.0，工程可控，适合和现有前端技术栈结合 | 需要自己写选择器、限速、存储和错误处理 | Apache-2.0 |
| Scrapy | Python 大规模结构化抓取 | 成熟、稳定、生态强 | 对重前端页面需要额外配合 Playwright/Splash | BSD-3-Clause |
| Playwright | 单页交互、登录态浏览、截图、手动样本采集 | 能模拟真实浏览器 | 不是完整爬虫框架，需要自己搭任务队列和存储 | Apache-2.0 |

## 3. Firecrawl 是否“最好”

不能简单说 Firecrawl 是所有场景最好，但可以说：

- 如果目标是“给 AI 应用拿网页内容”，Firecrawl 是目前非常强的选择。
- 如果目标是“快速验证公开网页样本采集”，Firecrawl 很适合先试。
- 如果目标是“长期稳定采集小红书样本”，Firecrawl 未必够，需要实际测试。
- 如果目标是“商业产品内置爬虫能力”，Firecrawl 自托管许可证需要谨慎评估。

## 4. 为什么小红书要谨慎

小红书样本采集有几个特殊点：

- 页面内容强依赖前端渲染。
- 登录状态影响可见内容。
- 公开搜索引擎收录不稳定。
- 互动数据可能随时间变化。
- 平台对自动化访问敏感。

因此 MVP 阶段建议只做：

- 小批量
- 公开内容
- 明确关键词或指定链接
- 低频采集
- 不自动点赞、收藏、评论、关注、私信
- 不绕过验证码、风控或权限限制

## 5. 推荐 MVP 样本采集流程

### 5.1 输入

产品经理或用户提供：

- 民宿名，例如“大理米糖的院子”
- 城市关键词，例如“大理民宿”
- 场景关键词，例如“古城小院”“亲子民宿”“宠物友好”
- 小红书笔记链接，如有

### 5.2 采集

优先使用 Firecrawl：

- 搜索关键词
- 抽取公开网页内容
- 输出标题、正文、标签、互动数据、链接

如果 Firecrawl 失败：

- 改用 Crawlee + Playwright
- 或人工打开页面后保存可见样本

### 5.3 结构化

每条样本整理为：

```json
{
  "source_url": "",
  "city": "",
  "scene": "",
  "account_type": "",
  "title": "",
  "opening": "",
  "body_structure": "",
  "selling_points": [],
  "hashtags": [],
  "image_structure": [],
  "likes": null,
  "collects": null,
  "comments": null,
  "consultation_signals": [],
  "reusable_patterns": [],
  "risk_patterns": []
}
```

### 5.4 入库

先不用数据库，MVP 阶段可以写入：

- `docs/真实民宿样本库.md`
- 后续再迁移为 JSON / PostgreSQL

## 6. 建议下一步

### 方案 A：先用 Firecrawl 云服务

适合最快验证。

需要：

- 注册 Firecrawl
- 获取 API Key
- 写一个本地脚本：输入关键词或 URL，输出结构化样本

优点：

- 快
- 少写爬虫细节
- 对 AI 工作流友好

风险：

- 需要 API 费用
- 小红书成功率未知
- 云服务会访问目标网页，合规上要谨慎

### 方案 B：用 Crawlee + Playwright 自建

适合后续可控迭代。

需要：

- 安装 Crawlee
- 用 Playwright 打开公开搜索页
- 抽取可见标题、链接、互动数字
- 限速、去重、错误重试

优点：

- Apache-2.0
- 和现有 TypeScript 技术栈更一致
- 可控性强

风险：

- 工程成本更高
- 小红书页面变化后要维护选择器

## 7. 当前建议

短期建议：

先试 Firecrawl，不要急着自建复杂爬虫。

验证目标：

- 能否通过关键词找到小红书相关公开结果。
- 能否抽取标题、正文、标签和互动数。
- 能否稳定处理指定笔记链接。

如果 Firecrawl 对小红书效果不好，再转 Crawlee + Playwright。

如果两者都不稳定，则样本库第一阶段采用“人工指定链接/截图 + AI 拆解”的半自动流程。

## 7.1 无 Firecrawl API Key 时的免费路线

如果只是验证，不想花钱，推荐直接跳过 Firecrawl 云服务，使用：

```text
Crawlee + Playwright 本地采样
```

原因：

- Crawlee 是开源爬虫框架，Apache-2.0，适合商业项目继续使用。
- Playwright 是开源浏览器自动化工具，Apache-2.0。
- 本地运行不需要购买 Firecrawl credits。
- 对小红书这种重前端页面，真实浏览器路线通常比普通 HTTP 抓取更接近真实用户看到的内容。

限制：

- 需要安装依赖，工程成本比 Firecrawl 高。
- 仍然不能保证小红书稳定可抓。
- 只能做小批量公开样本验证。
- 不做自动点赞、收藏、评论、关注、私信。
- 不绕过验证码、风控、登录限制或权限限制。

推荐验证目标：

- 每次只采 5-10 条搜索结果。
- 关键词先用“大理民宿”“大理古城民宿”“大理小院民宿”“米糖的院子”。
- 只保存标题、链接、可见互动数、正文可见文本和图片结构描述。
- 后续由 AI/人工拆解成样本库。

如果这个免费方案效果不好，再考虑 Firecrawl、Apify 云端或人工样本补充。

## 8. 本地试用脚本

已新增脚本：

```text
app/scripts/firecrawl-sample.mjs
```

配置：

```env
FIRECRAWL_API_KEY=你的 Firecrawl API Key
```

用法：

```bash
node scripts/firecrawl-sample.mjs --query "大理米糖的院子 小红书" --limit 5
node scripts/firecrawl-sample.mjs --query "大理民宿 小红书" --limit 5 --scrape true
node scripts/firecrawl-sample.mjs --url "https://example.com/page"
```

输出目录：

```text
samples/firecrawl/
```

输出包括：

- 原始 JSON
- 便于人工拆解的 Markdown

注意：

- `--scrape true` 会消耗更多 Firecrawl credits。
- 小红书页面可能无法稳定抽取，这是本次验证的重点。
- 不建议大规模跑关键词，先小批量验证 3-5 个关键词或链接。
